# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Model Flow is a CLI + Textual-based GUI for orchestrating modular simulation models (e.g. IFM-CAP) built from many independent R / R Markdown / GAMS / Windows batch scripts. Scripts declare their identity and configuration via inline `@MODELFLOW_*` annotation comments; `model_flow` scans a code directory, parses those annotations into a JSON "database" (`model_flow.db.json`), and can then run individual tasks (and eventually pipelines) using that metadata.

Terminology (also in README.md): a **Module** is a folder containing **Tasks** (one script each). A **Pipeline** is an ordered sequence of tasks within a module, declared per-module in a `model_flow.pipelines.json` source file and executed via `ExecutionEngine.execute_pipeline` (see below) — stop-on-first-failure, in declared order. Each pipeline task entry can optionally carry static parameter overrides and/or a loop over one or more parameters driven by named **Lists** (`classes/Lists.py`), run sequentially or in parallel — see `classes/Parser.py`'s `parse_pipelines`/`expand_loop` and `classes/Database.py`'s pipeline CRUD methods below. A **Workflow** is an ordered sequence of modules (not yet implemented). GUI support for pipelines (browsing, running, editing a non-looped task's parameters for one run) already exists in `textual_gui/app.py`; only *authoring new pipeline definitions* (or editing a loop's own declaration) from within the GUI is not yet implemented — pipelines are still hand-authored as `model_flow.pipelines.json`.

## Running the tool

Install runtime deps with `pip install -r requirements.txt` (`textual`, `rich`, `numpy`), or `pip install -r requirements-dev.txt` to also get `pytest`/`pytest-asyncio`.

```bash
python model_flow.py init                                   # interactively create model_flow.config.json
python model_flow.py build --config=<config.json|dir>        # scan Code_directory, write model_flow.db.json
python model_flow.py list_tasks --config=<config> [--module=<name>]
python model_flow.py show_task --config=<config> --module=<name> --task=<name>
python model_flow.py run_task --config=<config> --module=<name> --task=<name> [--set VAR=VALUE ...] [--parallel] [--range VAR START END STEP] [--values VAR V1 V2 ...] [--output_dir <dir>]
python model_flow.py run_pipeline --config=<config> --module=<name> --pipeline=<name> [--output_dir <dir>]   # sequential, stop-on-failure; no --set/--parallel yet
python model_flow.py run_gui [--config=<config>]              # launches textual_gui/app.py
```

`--config` accepts either a path directly to a config JSON file, or a directory containing `model_flow.config.json`.

Run the test suite with `python -m pytest test/`. Tests live in `test/test_*.py` (pytest auto-discovered). `test/fixtures/` holds pre-existing hand-written fixture files (`test_script.R`, `.rmd`, `.gms`, `.bat`, a sample `ifmcap_flow.config.json`, `module_flow.json`) used for manual exercising rather than by pytest — note those legacy fixtures still use the `@IFMCAP_*` annotation prefix, not the current `@MODELFLOW_*` one, so they won't parse against the current `Task` parser as-is. Coverage is deliberately unit-level: `Task`/`Parser` annotation parsing against in-memory fixture files, `ExecutionEngine`/`Database` behavior with `subprocess.call`/`subprocess.run` mocked via `monkeypatch`, and `textual_gui/app.py`'s wiring exercised headlessly via Textual's `Pilot` (`App.run_test()`, tests marked `@pytest.mark.asyncio` / `test/test_gui.py`) — there's no coverage that actually invokes `Rscript`/`GAMS`, since the configured executable paths are machine-specific and won't exist in most dev/CI environments. When adding coverage for a new code path, follow this same pattern rather than trying to run real scripts.

## Architecture

Data flows: **annotated script → `Task` (parses one file) → `Parser.parse_modules` (walks a directory, builds one `Task` per script) → `model_flow.db.json` (via `build` command) → `Database` (loads/queries that JSON) → `ExecutionEngine` (runs a task's underlying script)**. The CLI (`model_flow.py`) and the GUI (`textual_gui/app.py`) are two front ends over the same `Database`/`ExecutionEngine`/`Config` classes.

- **`classes/Config.py`** — loads/validates `model_flow.config.json` (or an inline JSON string). Required keys: `Code_directory`, `Database_directory`, `Temporary_directory`, `Rscript_exe`, `GAMS_exe`. `Config.get()` is case-insensitive. `is_empty()` signals a failed/invalid load — callers must check it (constructor swallows errors into `self.data = None` rather than raising). `create_from_user_input()` (used by the `init` CLI command) validates each path as it's entered — `DIRECTORY_KEYS` must be an existing directory, `EXECUTABLE_KEYS` (`Rscript_exe`, `GAMS_exe`) must be an existing *file* (not just its containing directory — a directory here is a common, previously-silent misconfiguration that surfaces as a cryptic error deep in `ExecutionEngine`), `DIRECTORY_CREATE` (`Temporary_directory`) is auto-created if missing. An invalid entry prints a warning and re-prompts for that same key rather than aborting the whole flow. Note `Pandoc_dir` (used by `ExecutionEngine._execute_rmd_task`) is *not* one of the required/prompted keys — it's optional at the `Config` level but effectively required for any `.rmd` task, so a config built via `init` can still be missing it; `ExecutionEngine._config_path()` raises a clear `ValueError` if so, rather than a bare `Path(None)` `TypeError`.

- **`classes/Task.py`** — every annotation-line and config-value-line regex is a named class-level constant (`ANNOTATION_PATTERN_R`/`_GAMS`/`_BAT`, `CONFIG_VALUE_PATTERN_R`/`_RMD`/`_GAMS`/`_BAT`, `ATTRIBUTE_PATTERN`) rather than an inline literal — this is so `annotation-spec.json` (repo root) can mirror them for `vscode-extension/`'s live diagnostics/snippets/hover, with `test/test_annotation_spec_matches_task_py.py` asserting the two never drift apart. If you change any of these patterns, update `annotation-spec.json` too (that test will fail otherwise) — see `vscode-extension/README.md`. Parses a single script file for `@MODELFLOW_*` annotations. Three parsers: `_parse_file_R` (`.r`/`.rmd`, annotation prefix `#@MODELFLOW_...`), `_parse_file_GAMS` (`.gms`, annotation prefix `*@MODELFLOW_...`), and `_parse_file_BAT` (`.bat`, annotation prefix `::@MODELFLOW_...` — `::` because cmd.exe always treats it as a no-op, unlike `REM` which needs a trailing space to parse as a comment). Recognized annotations: `task` (sets `name`/`module`), `config` (appends an entry to `self.config`, capturing the *next line's* literal assignment as `script_name`/`script_value` — this is how default parameter values are discovered without executing the script), and `description_start`/`description_end` (accumulates free text into `self.description`). A file with no `task` annotation yields `self.name = False` and is skipped by the parser. `_parse_file_BAT`'s config-line regex requires the exact guarded, quoted form `IF NOT DEFINED VAR SET "VAR=value"` (guard and `SET` variable names must match, case-insensitively) — a bare unquoted `set VAR=value`, the old unguarded `SET "VAR=value"`, a guard/`SET` referring to different variable names, or anything else does not match and is **rejected**: the parser prints a warning (`Warning: invalid MODELFLOW_config parameter ...`) and skips that config entry entirely (it never gets appended to `self.config`), rather than silently recording something malformed. The guard requirement is deliberate, not an oversight: it's what makes `--set`/GUI overrides actually take effect for `.bat` tasks (see `ExecutionEngine.py` below) — don't widen the regex back to accept the unguarded form without checking whether that's actually wanted first.

- **`classes/Parser.py`** — `Parser.parse_modules(directory)` walks a directory tree (skipping `.git`, `.vscode`, `.svn`, `__pycache__`, `venv`), constructs a `Task` for every `.r`/`.rmd`/`.gms`/`.bat` file (`Parser.allowed_extensions`), and groups the resulting task dicts by `module` name into `{module_name: [task_dict, ...]}`. This is the dict that gets serialized as `model_flow.db.json` by the `build` CLI command. Also has `parse_range_args`/`parse_value_args`, which turn `--range`/`--values` CLI args into value lists.
  - `Parser.parse_pipelines(directory, modules, lists=None, on_file=None)` — a **second**, independent walk (run by `build` right after `parse_modules`/`parse_lists` — note `parse_lists` now runs *before* `parse_pipelines`, since validating a loop's list references needs `lists` already built), looking for files literally named `Parser.pipelines_filename` (`"model_flow.pipelines.json"`) rather than matching an extension. It's a separate pass rather than folded into `parse_modules`'s own walk because validating a pipeline's task list needs the *complete* `modules` dict (module identity is global to the whole directory, not tied to a single folder), which only exists once `parse_modules` has already finished regardless. Each pipelines file must declare `"module"` (must match a module name some `Task` actually produced) and a `"pipelines"` list of `{"name", "tasks", "description"?}` objects, where `tasks` is an ordered list of steps — each either a **task name** string (matching `Task.name`, not filenames) or an object `{"task", "overrides"?, "loop"?}` — that must all reference tasks belonging to that same module — a pipeline cannot span modules. `_normalize_pipeline_tasks` (a private helper) validates and **normalizes every entry, string or object, into `{"task", "overrides", "loop"}`** before it's stored — this is the shape everything downstream (`Database`, `ExecutionEngine`, the GUI) consumes, so nothing else needs to branch on how a step was authored. `overrides` keys and `loop["parameters"]` keys must each match a real `script_name` in that task's own parsed `config` (and must be disjoint from each other); `loop["parameters"]` values must each name a List present in `lists`; `loop["combine"]` (`"zip"` or `"product"`) is required only when looping over more than one parameter, and a `"zip"` loop's referenced Lists must all have equal `len(elements)` (checked directly, not via a full `expand_loop` expansion — a `"product"` loop's cartesian combination could be enormous and there's no need to materialize it just to validate lengths). Validation is warn-and-skip at the finest granularity (mirrors `Task.py`'s `.bat`-config warning pattern): malformed JSON or a missing `module` skips the whole file; a missing/duplicate `name`, an empty/malformed `tasks` list, or *any* problem with a single task entry (unknown task, unknown/overlapping override or loop-parameter key, unknown list, mismatched `"zip"` lengths, bad `combine`/`mode`) skips that **whole pipeline entry** — deliberately the same coarse granularity as an unknown task name always had, not a new finer one. Duplicate pipeline names for the same module are tracked (and rejected) across the *entire* walk, not just within one file, since multiple folders can declare the same module.
  - `Parser.expand_loop(loop, resolve)` — pure static helper turning a validated `loop` dict into an ordered list of per-iteration `{script_name: value}` override dicts. `resolve(list_name)` is a plain callable returning that list's elements — `lambda name: lists[name]["elements"]` at parse/build time, `Lists.get_elements` at run time (re-resolving current list contents rather than trusting a build-time snapshot, since a list's elements can change independently of when the pipeline itself was last built). A single-parameter loop just iterates that one list's elements; multiple parameters combine via `zip` (pairwise) or `itertools.product` (full cartesian) per `loop["combine"]`.
  - `Parser.parse_lists(directory)` — a **third** independent walk (also run by `build`), looking for files literally named `Parser.lists_filename` (`"model_flow.lists.json"`). Unlike pipelines, lists have no module scoping, so a `model_flow.lists.json` can sit in *any* folder of `directory`, not just one per module — the walk doesn't need `parse_modules`'s output at all. Each file declares a `"lists"` list of `{"name", "type", "elements", "description"?}` objects; every collected entry is tagged with a `"folder"` key (the entry's source folder, relative to `directory`, forward-slash-joined, `"."` for the root itself) so that provenance survives once every file's lists are merged into one flat `{list_name: entry}` dict. A list `name` must be unique across the *entire* walk (not per-folder) since the merged result and `classes/Lists.py` both key purely by name — a duplicate found in another folder is warn-and-skipped, first-seen wins (same pattern as pipelines' cross-folder duplicate-name handling, just without the module dimension).

- **`classes/Database.py`** — thin JSON-backed store wrapping `model_flow.db.json` (path derived from `Config.Database_directory`). CRUD-ish helpers: `list_modules`, `list_module_tasks`, `get_module`, `get_task`, `add_module`, `add_task`, `delete_task`, `save`. Constructor raises `FileNotFoundError` if the db file hasn't been `build`-generated yet. It also owns a second, optional file, `model_flow.db_user.json` (same directory) — GUI-only, never read by the CLI/`ExecutionEngine.execute_task` — holding a *history* of user-entered parameter values per task, keyed by task `name` (not `file`, to match `get_task`'s own key): `{module: [{"name": task_name, "config": [{"name": script_name, "script_value": [older, ..., newest]}]}]}`. `get_user_values(module, task_name)` returns `{script_name: [history...]}`; `add_user_value(module, task_name, script_name, value)` appends (skipping consecutive duplicates) and persists. Missing user-db file is not an error (defaults to `{}`), unlike the main db.
  - **Pipelines**: a third pair of files, `model_flow.pipelines.json` (the `build`-aggregated output of `Parser.parse_pipelines`, mirrors `model_flow.db.json` — every `tasks` entry already normalized to `{"task", "overrides", "loop"}` regardless of how it was authored) and `model_flow.pipelines_user.json` (user/future-GUI-authored pipelines, mirrors `model_flow.db_user.json` — kept as its own file rather than folded into `db_user.json`, since that file's shape is unrelated per-task parameter *history*, not pipeline definitions). Both are **optional** (default to `{}`, loaded only `if path.exists()`) — unlike `db_path`'s mandatory `FileNotFoundError` — deliberately, so that upgrading to a version with pipelines support doesn't break `run_task`/`run_gui` for a config that hasn't been rebuilt yet. `list_pipelines(module)`/`get_pipeline(module, name)` merge both files, with the source-declared (`pipelines_data`) pipeline winning on a name collision with a user-authored one. `add_pipeline(module, pipeline, user=False)`/`delete_pipeline(module, name, user=False)` take a `user` flag to pick which store is mutated: `user=False` (default) mutates `pipelines_data` in-memory only — caller must call `save_pipelines()`, matching `add_task`/`delete_task`'s own non-auto-persisting contract, since source data is `build`-regenerated anyway; `user=True` mutates `user_pipelines_data` and auto-persists immediately via `_save_user_pipelines()`, matching `add_user_value`'s contract.

- **`classes/ExecutionEngine.py`** — looks up a task via `Database.get_task`, then dispatches on `filetype` to `_execute_r_task`, `_execute_rmd_task`, `_execute_gams_task`, or `_execute_bat_task`, each of which runs the command via a shared `_run()` helper (R via `Rscript_exe`, `.rmd` via `Rscript_exe -e 'rmarkdown::render(...)'` using `Pandoc_dir`/`GAMS_exe` for env setup, GAMS via `GAMS_exe`, `.bat` via `["cmd", "/c", <script>]` — explicit `cmd /c` rather than invoking the `.bat` path directly, since a batch file isn't a PE executable and relying on it "just working" without `shell=True` is version/behavior-dependent; this also avoids `shell=True` entirely, so there's no shell-quoting/injection concern). For R/GAMS, task `config` entries (`script_name`/`script_value` pairs captured by `Task`) become command-line/params arguments passed to the underlying script. **`.bat` is different**: config entries are passed as environment variables (`_execute_bat_task` builds `env = os.environ.copy()` + `env[script_name] = script_value`, passed through `_run()`'s `env` parameter down to `subprocess.call`/`.run`/`.Popen`), not positional CLI args — confirmed empirically against real `cmd.exe` that its own argument tokenizer splits on `=` (not just whitespace) when populating `%1`/`%2`/`%*`, so a `NAME=value` token never survives as one argument, and the workaround (quoting) runs straight into a separate `cmd /c` quote-mismatch trap. Since `Task.py` now only accepts the guarded `IF NOT DEFINED VAR SET "VAR=value"` form (see above), the script's own `SET` only fires when the variable hasn't already been supplied via the environment — so a value `_execute_bat_task` injected beforehand survives, and `--set`/GUI overrides now take effect for `.bat` tasks the same as R/GAMS/Rmd. This was fixed purely by tightening `Task.py`'s annotation contract; `ExecutionEngine.py` itself needed no change, since it was already injecting overrides via `env` correctly. Note the asymmetric path handling: `_execute_r_task` and `_execute_rmd_task` both convert Windows backslashes to forward slashes via a local `r_path()` helper before handing paths to R, but `_execute_gams_task`/`_execute_bat_task` do not (GAMS and cmd.exe both accept/expect native Windows paths) — preserve this distinction if adding a new task-type executor. `.bat` needs no dedicated `Config` key (unlike `Rscript_exe`/`GAMS_exe`) since `cmd` is always resolvable on `PATH` on Windows.
  - `execute_pipeline(module, pipeline_name, output_dir=None, capture_output=False, on_output=None, on_process_start=None, on_step_start=None, extra_overrides=None)`: owns the whole pipeline-running algorithm (both `model_flow.py`'s CLI `run_pipeline` and `textual_gui/app.py`'s `_execute_selected_pipeline` are now thin callers, so loop/parallel logic lives in exactly one place). Constructed alongside `self.database` in `__init__` is `self.lists = Lists(config)`, used to resolve a loop's `parameters` list-names to live elements at run time (`Parser.expand_loop(loop, self.lists.get_elements)`), re-resolving current list contents rather than trusting whatever the pipeline's own last `build` snapshot saw. For each normalized `{"task", "overrides", "loop"}` entry (defensively, a bare task-name string from a stale pre-loop-feature `model_flow.pipelines.json` is treated the same as `{"task": name, "overrides": {}, "loop": None}` rather than crashing): no `loop` runs the task once; a `loop` expands to an ordered list of per-iteration override dicts (merged onto the entry's own `overrides`, then `extra_overrides.get(task_name, {})` — the GUI's per-run edits — as an additional base layer, both pre-validated/expected to not collide with the loop's own per-iteration keys), each iteration executed against its own output subdirectory (`_pipeline_iteration_output_dir`: `output_dir/task_name/k=v__...`, sanitized and created via `mkdir(parents=True, exist_ok=True)`) to avoid `_execute_rmd_task`'s minute-granularity output-filename collision across repeated runs of the same task. `mode: "sequential"` (default) stops the whole pipeline at the first non-zero iteration; `mode: "parallel"` submits every iteration to a `ThreadPoolExecutor` (`max_workers` from the loop, else `min(iteration_count, os.cpu_count() or 4)`), lets *all* of them finish (a sibling's failure doesn't cancel already-launched iterations — simpler than mid-flight cancellation, a deliberate tradeoff), then fails the step on the first non-zero result in submission order if any occurred. Always returns a plain `int` (0 or the first non-zero code) regardless of `capture_output`, since a pipeline's output is already delivered live via `on_output`/`on_step_start` rather than as one combined `ExecutionResult`. `on_step_start(step_index, total_steps, task_name, iteration_index, total_iterations, iteration_values)` fires once per individual execution (once for a non-looped task, once per iteration otherwise) — this is how a caller like the GUI renders live per-step/per-iteration progress without execute_pipeline needing to know anything about a UI. Raises `ValueError` if the module/pipeline isn't found (same message `model_flow.py`'s old inline loop used to raise itself).
  - `execute_task(module, task_name, output_dir=None, overrides=None, capture_output=False, on_output=None, on_process_start=None)`: `overrides` (a `{script_name: value}` dict) is applied to a `copy.deepcopy` of the fetched task — the underlying `Database`'s cached task dict is never mutated. `capture_output=False` (the default, used by the CLI) preserves the original behavior exactly: `subprocess.call()` inherits stdio, output streams live to the console, an `int` return code comes back. `capture_output=True` (used by the GUI) returns an `ExecutionResult(returncode, stdout, stderr)` dataclass instead — needed because `subprocess.call`'s inherited stdio would corrupt a Textual TUI's alt-screen rendering rather than showing up in any widget. Within capturing mode, `_run()` branches further: with no `on_output`, it's the original blocking `subprocess.run(capture_output=True)` (all output available only once the process exits); with `on_output` given, `_run_streaming()` uses `subprocess.Popen` + iterates `process.stdout` (stderr merged in via `stderr=subprocess.STDOUT`, so `ExecutionResult.stderr` is always `""` on this path) and calls `on_output(line)` for each line as it's produced, still returning the same `ExecutionResult` at the end with the joined lines. Caveat: this is only as "live" as the child process's own stdio buffering allows — R block-buffers stdout when it isn't a terminal, so scripts with no explicit flush may still arrive in bursts rather than truly per-line. `on_output` runs on whatever thread called `execute_task`; it is not made thread-safe here, that's the caller's responsibility. `on_process_start` (streaming path only) is called once with the live `subprocess.Popen` right after creation — this is how a caller gets a handle to `terminate()` the process from elsewhere (e.g. a GUI "kill" keybinding); terminating mid-stream just closes the stdout pipe, ending the read loop normally and still returning an `ExecutionResult` rather than raising. `_run()` also accepts an `env` dict (passed through to whichever `subprocess.*` call it makes) — `None` means "inherit the parent's environment" as usual; only `_execute_bat_task` supplies one (see below). When touching `_execute_*_task`, route new subprocess calls through `_run()` rather than calling `subprocess.call`/`subprocess.run`/`subprocess.Popen` directly, to keep this branching in one place.

- **`classes/Lists.py`** — standalone class, separate from `Database` purely for separation of concerns (not because of a directory split — both its files live in `Database_directory`, same as `Database`'s own files), that reads named value lists (e.g. NUTS0/NUTS2 region codes) for lookup by name. `model_flow.lists.json` is a **build-generated artifact** (the aggregated output of `Parser.parse_lists`, written by `build()`/GUI-rebuild alongside `model_flow.db.json`/`model_flow.pipelines.json`), not hand-maintained in `Database_directory` directly; its actual hand-maintained sources are the scattered `model_flow.lists.json` files anywhere under `Code_directory` that `Parser.parse_lists` collects. `model_flow.lists_user.json`, also in `Database_directory`, is the user-authored counterpart, mirroring `Database`'s `db_user`/`pipelines_user` split. Both files are optional. `list_names()`/`get_list(name)`/`get_elements(name)` merge both stores (shared wins over user on a name collision, same precedence as `Database.get_pipeline`); `add_list(entry, user=True)`/`delete_list(name, user=True)` take the same `user` flag contract as `Database.add_pipeline`/`delete_pipeline` (`user=True` default, auto-persists immediately; `user=False` mutates the shared in-memory store only — caller must call `save()`, since that store is `build`-regenerated anyway). `model_flow.py`'s `build()` and the GUI's `ctrl+b` (`action_rebuild_database`, below) both write it. `ExecutionEngine` now owns its own `Lists` instance (`self.lists`, constructed in `__init__`) purely to resolve a pipeline task's `loop.parameters` list-names to elements at run time (`execute_pipeline`, above) — the only consumer so far; still not read by `run_task`, or by the GUI's `ShowTask`/`SelectTask` browse views for anything beyond that.

- **`model_flow.py`** — argparse-based CLI entry point wiring the above together: `init`, `build`, `list_tasks`, `run_task`, `run_pipeline`, `show_task`, `run_gui`. `run_task` passes `--set`-parsed parameters through to `ExecutionEngine.execute_task` as `overrides` — this is the CLI's only override mechanism; it never reads `model_flow.db_user.json`. `--range`/`--values` are still parsed by argparse but not threaded through to execution (remaining known gap; unrelated to pipeline overrides/loops below, which are declared in the pipeline JSON itself, not via CLI flags). `build()` calls `Parser.parse_modules`, then `Parser.parse_lists`, then `Parser.parse_pipelines(str(code_dir), modules, lists)` — **`parse_lists` now runs before `parse_pipelines`** (order matters: validating a loop's list references needs `lists` already built) — and writes `model_flow.db.json`, `model_flow.pipelines.json`, and `model_flow.lists.json` directly via `json.dump` (not through `Database`/`Lists`, since both classes require their respective primary file to already exist — a pre-existing chicken/egg reason `build` never went through those classes even for tasks).
  - `run_pipeline(config, module, pipeline_name, output_dir=None)` is now a thin wrapper: constructs an `ExecutionEngine` and returns `engine.execute_pipeline(module, pipeline_name, output_dir)` directly — the actual per-task/per-iteration loop, override merging, and sequential/parallel loop execution all live in `ExecutionEngine.execute_pipeline` itself (see above), not here, so the CLI and the GUI's `_execute_selected_pipeline` share one implementation instead of each re-deriving it. Unlike `run_task`'s CLI wiring (which discards its int result), `run_pipeline`'s CLI dispatch does propagate a non-zero result to `sys.exit(...)` — a small, deliberate asymmetry versus `run_task`, not a retroactive fix applied there too.

- **`textual_gui/app.py`** — `ModelFlowApp` (Textual `App`) composes `SelectTask` (a searchable module/task tree, left pane), `ShowTask` (task detail + editable config, right pane), and `ExecuteTask` (status + output log, bottom pane) — unless `Database(config)` raised `FileNotFoundError` at startup, in which case `compose()` renders a `#startup-error` message directing the user to run `build` instead. Constructing `self.database`/`self.engine` (an `ExecutionEngine`) happens once in `__init__`, guarded by that same `try/except`.
  - `SelectTask`'s `#module-search` `Input` filters the tree live via `on_input_changed` → `filter_tree()` (case-insensitive substring match on module/task names).
  - Selecting a tree node calls `ModelFlowApp.select_task`, which fetches the task via `Database.get_task` and awaits `ShowTask.show_task(task)`. `ShowTask` renders one editable row per `config` entry (an `Input` prefilled with the base `script_value`, plus a `Select` populated from `Database.get_user_values` history when one exists — picking a history value fills the `Input`). `ShowTask.get_overrides()` diffs current `Input` values against the task's base defaults to build the same `{script_name: value}` shape `ExecutionEngine.execute_task` expects.
  - **Gotcha**: `ShowTask`'s tree/log widgets are named `task_tree`/`description_log`, not `tree`/`log` — `Widget` itself has built-in `tree`/`log` *properties* (devtools facilities, no setter), so `self.tree = Tree(...)` inside `__init__` raises `AttributeError: property 'tree' ... has no setter`. (`SelectTask.tree` avoids this only because `tree = None` is declared at the *class* body level, which shadows the inherited property outright — an instance-level assignment in `__init__` does not get that shadowing.) Avoid `self.tree`/`self.log` as instance attribute names anywhere in this file.
  - Pressing `ctrl+r` (bound in `ModelFlowApp.BINDINGS`, shown via the `Footer`) runs `action_execute_task`: gathers overrides from `ShowTask`, then runs `ExecutionEngine.execute_task(..., capture_output=True, on_output=on_output, on_process_start=on_process_start)` via `await asyncio.to_thread(...)` (subprocess calls block, so this keeps the UI responsive). `on_output` is a closure that does `self.call_from_thread(self.execute_panel.append_output, line)` — `execute_task` (and therefore `on_output`) runs on the thread `asyncio.to_thread` spawns, and Textual widgets are not thread-safe to touch directly from there, so every line must be marshaled back to the UI thread via `call_from_thread` (which itself raises `RuntimeError` if mistakenly called from the UI thread — a real bug class here, not a hypothetical one). `ExecuteTask.start_running()` clears `output_log` at the start of a run so it's ready to receive streamed lines; `show_result(..., streamed=True)` then must *not* rewrite `result.stdout`/`stderr` (they're already on screen, appended live) — only `streamed=False` callers get the old "write everything at the end" behavior. Either way, `show_result` always writes a final `"Task {module}/{task_name}, finished"` line to `output_log` after any streamed/replayed output, so completion is visible in the log itself, not just the status line above it. On completion, calls `Database.add_user_value` for each overridden parameter so it's offered as a history option next time.
  - `on_process_start` (a plain closure, `self.current_processes.append(process)` — no `call_from_thread` needed, since list mutation doesn't touch a Textual widget/reactive) stashes each live `subprocess.Popen` on the app so `ctrl+k` (`action_kill_task`) can `terminate()` it/them. `self.current_processes` is a **list**, not a single process, because a parallel pipeline loop step can have several subprocesses running at once (see below) — a single-task run just appends exactly one. `action_execute_task` refuses to start a second run while any tracked process is still alive (`any(p.poll() is None for p in self.current_processes)`) — `ctrl+k` first. `action_kill_task` sets `self.execution_cancelled = True` before `terminate()`-ing every currently-live tracked process; `action_execute_task`'s completion handling (both the normal-return and the `except Exception` paths, since a killed process can surface either way depending on timing) checks that flag first and calls the new `ExecuteTask.show_aborted(module, task_name)` instead of `show_result`/`show_error` when set — so a user-initiated kill reads as "aborted", not as a mysterious failure.
  - `ExecuteTask` and the browse view (`self.main_view`, the `Horizontal(SelectTask, ShowTask)`) are mutually exclusive, toggled via each other's `.display`, not stacked — `ExecuteTask`'s CSS is `height: 1fr` so it fills the full screen when shown. `ctrl+o` (`action_toggle_output`) flips between the two; `action_execute_task` always switches to the full-screen output view before running (`execute_panel.display = True`, `main_view.display = False`) so a result is never missed. Both start hidden/shown respectively (`execute_panel.display = False` set explicitly in `compose()`, since `Widget.display` otherwise defaults to `True`).
  - `ctrl+b` (`action_rebuild_database`) reruns `Parser.parse_modules`, then `Parser.parse_lists`, then `Parser.parse_pipelines(code_directory, modules, lists, on_file)` (this order matters, same reason as `model_flow.py`'s `build()` — see above) against `Code_directory` in a worker thread (each file scanned is streamed to the output log via an `on_file` callback, marshaled through `call_from_thread` same as `on_output`), writes the results via `self.database.save()`/`save_pipelines()`/`self.lists.save()`, and refreshes the browse tree (`SelectTask.refresh_from_database()`) — the in-GUI equivalent of the CLI `build` command. **Gotcha**: `ExecutionEngine` owns its *own* separate `Database`/`Lists` instances (constructed independently inside `ExecutionEngine.__init__`), not the same Python objects as `ModelFlowApp.database`/`self.lists`. A rebuild must explicitly also update `self.engine.database.data`/`pipelines_data` **and now `self.engine.lists.lists_data`** (needed so a pipeline loop referencing a just-rebuilt list resolves correctly without restarting the app), or execution keeps using stale pre-rebuild definitions even though the browse view and the files on disk are current.
  - **Pipeline browsing/running** (`ShowTask.show_pipeline`/`get_pipeline_overrides`, `ModelFlowApp._execute_selected_pipeline`): selecting a pipeline's tree node calls `show_pipeline(module, pipeline)`, which renders one section per `tasks` entry (defensively normalizing a bare string the same way `ExecutionEngine.execute_pipeline` does, for a stale un-rebuilt db file). A non-looped entry renders the usual editable `Input` row per config parameter, prefilled from `entry["overrides"]` layered over the task's own `script_value` default (not just the bare task default, so a JSON-declared override still shows as the starting value). A looped entry (`entry["loop"]` set) renders a **read-only** one-line summary instead (`ShowTask._describe_loop`, e.g. "Looped over nuts_code=nuts2 (parallel, up to 8 workers)") — there is no per-iteration editing UI; the loop itself can only be authored/changed in the JSON. `get_pipeline_overrides()` is unchanged in shape (`{task_name: {script_name: value}}`, built only from the non-looped rows that exist) and is passed to `ExecutionEngine.execute_pipeline` as `extra_overrides` — for a looped task any GUI edit would only apply if it doesn't collide with that iteration's own loop-driven value, same precedence as the JSON's own `overrides`. `_execute_selected_pipeline` no longer loops per-task itself: it makes one `asyncio.to_thread(self.engine.execute_pipeline, ...)` call for the *entire* pipeline, passing `on_output`/`on_process_start`/`on_step_start` closures (all marshaling through `call_from_thread`, since they now run on `execute_pipeline`'s single worker thread rather than the previous per-task `await asyncio.to_thread(...)` loop that gave control back to the event loop between tasks) and tracks `progress = {"task_name", "total_steps"}` updated by `on_step_start` purely so the final status line can still say "failed at task_a"/"succeeded (N tasks)" the way the old per-task loop's messages could trivially do.
  - `ExecuteTask.start_running(message)` sets the status text and `self.app.sub_title` to the same message (shown in the `Header`, at the top of the whole screen, regardless of whether `ExecuteTask` itself is currently toggled on/off via `ctrl+o`) — added because `ctrl+o` mid-run is easy to lose track of otherwise. `finish_running()` is the counterpart that clears the sub-title; every code path that ends a run (`show_result`, `show_error`, and the success tail of `action_rebuild_database`, which doesn't go through `show_result` since its "N modules, M tasks" summary isn't a task returncode) must call it — a test previously caught a gap here (the rebuild success path calling `set_status(...)` directly, which skipped it). There used to also be a `LoadingIndicator` spinner here, removed once output streaming (below) made it redundant — don't reintroduce one without checking this is still true.

- **`vscode-extension/`** — a separate TypeScript/Node subproject (its own `package.json`/`node_modules`/`tsconfig.json`, no relation to the Python side's dependencies) providing editor assistance for authoring `@MODELFLOW_*` annotations: commands that insert a correctly-formed `task`/`config`/`description_start`-`_end` block (with linked snippet tabstops so a config's name and its script variable name can't drift apart, `src/snippets.ts`), live diagnostics re-implementing `Task.py`'s own annotation/value-line walk (`src/diagnostics.ts`), and hover text (`src/hoverProvider.ts`). All three are driven by `annotation-spec.json` (repo root) — see the `Task.py` bullet above for how that spec is kept from drifting out of sync. `scripts/copy-spec.js` copies the root spec into `vscode-extension/annotation-spec.json` as an `npm run compile`'s `precompile` step, so the packaged extension ships its own copy rather than depending on the rest of the repo being present at the installed location; `src/annotationSpec.ts` reads that copy at runtime via `context.extensionPath`. ## Annotation format (for editing/authoring task scripts)

`{C}@MODELFLOW_{annotation} [{attribute}="{value}"]*` where `{C}` is the language's comment character (`#` in R/Rmd, `*` in GAMS, `::` in `.bat`). Full annotation reference and worked GAMS/Rmd/Bat examples are in README.md ("Annotations" and "How to prepare files for model_flow" sections) — read those before modifying `Task.py`'s parsing regexes, since the two are tightly coupled (GAMS annotations are matched against `^\*\s*@MODELFLOW_(\w+):?\s*(.*)$`, R/Rmd against `^\s*#@MODELFLOW_(\w+)\s*(.*)$`, `.bat` against `^::\s*@MODELFLOW_(\w+):?\s*(.*)$`, and the following line's literal is regex-matched differently per filetype to extract `script_name`/`script_value` — `.bat`'s is `^if\s+not\s+defined\s+(\w+)\s+set\s+"(\w+)=(.*?)"\s*$` case-insensitive, guarded quoted form only, with the guard and `SET` variable names required to match — no other variant is accepted).
